Oracle APEX – arhitektura izvajanja kode

Pregled tega, kako APEX obdeluje uporabniške akcije, kako deluje Submit Page, ter kakšna je vloga komponent Computations, Validations, Processes in Branches.

Show Page

APEX pripravi in izriše stran za prikaz v brskalniku.

Accept Page

APEX sprejme oddane vrednosti in izvede strežniško logiko.

Session State

Vrednosti page itemov se na strežniku hranijo v session state.

REQUEST

Tipično vsebuje ime gumba, ki je sprožil submit.

Podrobnosti

Execution Flow

Osnovni model delovanja APEX-a lahko razdeliš na dva glavna dela: Show Page in Accept Page.

1
Show Page
APEX prebere definicijo strani, regionov, itemov, gumbov in procesov, nato generira HTML, CSS in JavaScript za brskalnik.
2
Uporabnik dela v brskalniku
Vnaša podatke, klika gumbe, sproža Dynamic Actions in JavaScript.
3
Submit ali AJAX klic
Če pride do Submit Page, APEX preide v polni strežniški cikel. Če gre za AJAX Dynamic Action, se izvede samo delna strežniška obdelava.
4
Accept Page
APEX shrani submitted vrednosti v session state in nato izvede computations, validations, processes ter branches.
5
Redirect ali ponovni prikaz strani
Če validacije uspejo, lahko APEX izvede branch in preusmeri na drugo stran. Če validacije padejo, ponovno prikaže isto stran z napakami.
Key Components

Najpomembnejše komponente v APEX page processing modelu so:

  • Buttons – sprožijo submit, redirect ali dynamic action.
  • Dynamic Actions – odjemalske ali AJAX akcije ob dogodkih.
  • Session State – strežniška hramba vrednosti itemov.
  • Computations – nastavitev vrednosti itemov.
  • Validations – preverjanje pravilnosti vhodnih podatkov.
  • Processes – dejanska poslovna logika, DML, PL/SQL, klici procedur.
  • Branches – odločitev, kam gre uporabnik po obdelavi.
  • REQUEST – identifikacija akcije, običajno ime gumba.
Pomembno: APEX ni samo “HTML aplikacija”, ampak predvsem request/response engine nad Oracle bazo, ki uporablja session state kot most med brskalnikom in strežnikom.
What Submit Page does

Submit Page pomeni, da se trenutne vrednosti s strani pošljejo na strežnik in APEX začne Accept Page fazo.

  1. Brskalnik pošlje vrednosti page itemov na strežnik.
  2. APEX te vrednosti zapiše v session state.
  3. Nastavi se REQUEST, pogosto na ime gumba.
  4. Zaženejo se submit-time computations.
  5. Izvedejo se validations.
  6. Če validacije uspejo, se izvedejo processes.
  7. Na koncu se preverijo branches in določi naslednja navigacija.
Submit Page sam po sebi še ne pomeni shranjevanja v tabelo. Shranjevanje se zgodi šele, če obstaja ustrezen process, ki izvede INSERT, UPDATE, MERGE ali klic procedure.
Computations

Computations so namenjene temu, da APEX nastavi vrednost nekega itema ob določenem trenutku izvajanja.

  • lahko nastavijo default vrednost ob renderiranju strani,
  • lahko izračunajo novo vrednost po submitu,
  • običajno so manjši, ciljno usmerjeni koraki.

Primer: P10_TOTAL := P10_QTY * P10_PRICE

Computation običajno ni “glavna poslovna logika”, ampak bolj izračun ali nastavitev vrednosti itema.
Validations

Validations preverjajo, ali je stanje podatkov pravilno, preden se izvedejo procesi.

  • required field
  • pravilen format številke ali datuma
  • logično pravilo med več polji
  • custom PL/SQL preverjanje

Če validacija ne uspe, APEX praviloma ne nadaljuje v glavne procese, ampak ponovno prikaže isto stran in pokaže napake.

Processes

Processes so del, kjer se običajno zgodi glavno delo na strežniku.

  • INSERT, UPDATE, DELETE
  • klic PL/SQL procedure ali package procedure
  • Automatic Row Processing
  • Interactive Grid DML processing
  • klic API-ja
  • čiščenje session state
Če želiš razumeti, “kaj stran v resnici naredi”, so procesi pogosto najpomembnejši del.
Branches

Branch določa, kam se uporabnik preusmeri po obdelavi.

  • na drugo APEX stran,
  • na isto stran,
  • na zunanji URL,
  • pogojno glede na REQUEST ali druge pogoje.

Pogosto branch dela nekaj takega: po uspešnem shranjevanju preusmeri na report stran ali nazaj na detail.

Dynamic Actions vs Submit Page

Dynamic Actions in Submit Page nista ista stvar.

  • Execute JavaScript Code deluje samo v brskalniku.
  • Execute Server-side Code naredi AJAX klic na strežnik, vendar to še ni poln page submit.
  • Submit Page sproži polni Accept Page lifecycle.
Zato se pogosto zgodi, da PL/SQL v Dynamic Action ne vidi najnovejših vrednosti, če niso navedene v Page Items to Submit.
REQUEST and button logic

Ko uporabnik klikne gumb z akcijo Submit Page, APEX običajno nastavi REQUEST na ime gumba.

To omogoča, da ima ista stran več različnih poti izvajanja:

  • SAVE
  • DELETE
  • CANCEL
  • APPROVE

Posamezni validations, processes in branches se lahko izvajajo samo za določen REQUEST.

Practical interpretation

Če želiš stran v APEX-u razumeti arhitekturno, jo glej tako:

  1. Kaj se zgodi ob renderiranju?
  2. Kateri gumbi sprožijo submit, redirect ali dynamic action?
  3. Kateri itemi se pošiljajo v session state?
  4. Katere validations odločajo, ali se nadaljuje?
  5. Kateri processes izvajajo dejansko poslovno logiko?
  6. Kateri branch določi naslednji korak?
Zelo poenostavljeno: Computation = nastavi vrednost, Validation = preveri, Process = naredi delo, Branch = odloči, kam naprej.
